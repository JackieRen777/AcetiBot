# 证据门禁与自验收执行计划（2026-07-07）

## 背景

当前系统已经具备：

- 子库路由
- 分层检索
- 基础引用
- 部分表格事实抽取

但仍存在 4 个核心盲点：

1. 回答准入规则主要停留在 Prompt，没有完全变成程序规则。
2. 缺少明确的拒答门槛，导致“有点像相关”时仍可能继续作答。
3. 固定回归题库不足，难以验证每次修改是否提升了可信度。
4. `paper` 子数据库后续接入时，缺少预留的路由、证据和验收位。

## 总目标

把系统从“能回答”推进到“能判断自己什么时候不能回答”。

## 原则

### 1. 先准入，后生成

只有通过证据门禁的问题，才允许调用大模型生成答案。

### 2. 先证据匹配，后组织语言

回答是否允许，不由模型主观判断，而由：

- 问题类型
- 命中的子库
- 命中的证据粒度
- 关键指标词是否出现
- 是否满足最低证据数量

共同决定。

### 3. 拒答不是失败，而是质量控制

当证据不足、证据错位或证据粒度不够时，系统应明确拒答，并说明缺什么资料。

## 分阶段计划

## Phase 1：程序化证据门禁

### 目标

把“能不能答”变成可执行代码规则。

### 任务

1. 在 `query.py` 中增加 `evidence_gate` 评估函数。
2. 为不同问题类型定义最低证据要求：
   - `consumer_insight`
   - `standard_compliance`
   - `patent_process`
   - `flavor_mechanism`
   - `formula_process`
3. 对以下情况强制拒答：
   - 没有命中对应子库
   - 关键指标问题未命中关键指标词
   - 配方建议题没有主证据层
   - 用户明确要求上传数据，但系统未收到数据
4. 在普通查询与流式查询中统一接入门禁结果。

### 自验收

- 问题无命中时，系统是否拒答而不是泛泛作答
- 指标问题命中错库时，系统是否拒答
- 配方题缺主证据时，系统是否拒答

## Phase 2：固定回归题库

### 目标

让每次迭代都能重复验证“是否比上一次更可信”。

### 任务

1. 新建本地验收题库文件：
   - 问题
   - 期望路由
   - 是否应拒答
   - 期望命中的文档类型或来源
   - 备注
2. 新建验收脚本：
   - 跑 `prepare_query`
   - 检查路由是否正确
   - 检查门禁是否符合预期
   - 检查首屏证据是否基本合理
3. 输出结构化验收结果。

### 自验收

- 至少覆盖：
  - 消费者统计题
  - 标准合规题
  - 风味机理题
  - 配方建议题
  - 必须拒答题

## Phase 3：高风险资料控噪

### 目标

降低复杂论文表格和老旧扫描专利对回答可靠性的污染。

### 任务

1. 对高噪声 `flavor` 论文增加页级或表级门禁。
2. 对老旧扫描专利增加“解析失败即排除”的稳定策略。
3. 对标准大文件增加分片与去重策略。

### 自验收

- `哪组样品总酸更高？` 这类题不再被无关表格事实占据首位
- 配方建议题首屏证据更偏向可解释的段落与实施例

## Phase 4：paper 子数据库接入预留

### 目标

为你后续上传的 `paper` 子数据库留好插槽，而不是上传后再重构。

### 任务

1. 在路由语义上明确：
   - `paper` 是工艺主证据层的一部分
   - 在 `formula_process` 中优先级高于 `consumer`
2. 在验收题库中预留 `paper` 类型案例。
3. 在文档中明确未来的解析重点：
   - 论文正文段落
   - 表格参数
   - 方法条件
   - 结论段

### 自验收

- 后续 `paper` 上传后，只需补跑题库即可确认是否接入成功

## 本轮执行优先级

1. `Phase 1`
2. `Phase 2`
3. 补一轮本地验收记录
4. 暂不做全量正式重建

## 本轮完成标准

满足以下条件，才算这一轮完成：

1. 查询链路已接入程序化证据门禁
2. 系统可以对一部分问题明确拒答
3. 已存在可重复运行的本地验收题库
4. 已输出一份新的验收结果记录

## 备注

- 你后续上传 `paper` 子数据库时，本计划直接进入 `Phase 4` 扩展，不需要推翻前面结构。
- 全量正式入库和上线，放在本地门禁与验收稳定之后再做。
